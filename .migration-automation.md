# User Migration Automation

## Summary

Created an automated workflow for migrating users from Better Auth to Meteor Accounts collections.

## Files Created

### 1. `/run-migration.sh`
Main automation script that orchestrates the entire migration workflow.

**Features:**
- Lists available MongoDB databases
- Prompts for database selection (or reads from `TARGET_DB` env var)
- Runs dry-run preview first
- Asks for user confirmation
- Performs actual migration
- Runs verification automatically
- Clear error handling with immediate exit on failure
- Color-coded output for better UX

**Usage:**
```bash
# Interactive mode
./run-migration.sh

# With environment variable
TARGET_DB=meteor ./run-migration.sh
```

**Covers:**

## How It Works

### Workflow Steps

1. **Database Discovery**
   ```bash
   mongosh mongodb://localhost:27017 --eval 'db.getMongo().getDBNames()'
   ```
   Lists all databases in the local MongoDB instance.

2. **User Input**
   - If `TARGET_DB` env var is set, uses that
   - Otherwise, prompts user to type database name
   - Validates database exists in the list (with option to proceed anyway)

3. **Dry Run Preview**
   ```bash
   MONGO_URL="mongodb://localhost:27017/DATABASE" \
   node meteor-backend/scripts/migrate-to-meteor-accounts.js --dry-run
   ```
   Shows what would be migrated without making changes.

4. **Confirmation**
   - Pauses for user to review dry-run output
   - Asks "Proceed with actual migration? (y/N)"
   - Exits if user says no

5. **Actual Migration**
   ```bash
   MONGO_URL="mongodb://localhost:27017/DATABASE" \
   node meteor-backend/scripts/migrate-to-meteor-accounts.js
   ```
   Performs the actual migration.

6. **Verification**
   ```bash
   MONGO_URL="mongodb://localhost:27017/DATABASE" \
   node meteor-backend/scripts/verify-user-collections.js
   ```
   Verifies migration results and reports any discrepancies.

### Environment Variables

The scripts use `MONGO_URL` which includes both host and database:
```bash
export MONGO_URL="mongodb://localhost:27017/DATABASE_NAME"
```

The automation script constructs this from:
- `MONGO_HOST` (default: `mongodb://localhost:27017`)
- User-provided or `TARGET_DB` environment variable

## Migration Details

### What Gets Migrated

- Email (required, used as unique key)
- Email verification status
- Name → profile.name
- Username (only if not null, to avoid unique constraint issues)
- Image URL
- Bio
- Website
- Reports-to relationship
- Blocked users array
- Created date

### What Does NOT Migrate

- **Passwords** - Better Auth passwords cannot be decrypted
- All users must reset their passwords after migration

### Schema Conversion

**Better Auth (`user` collection):**
```javascript
{
  _id: ObjectId("..."),
  email: "user@example.com",
  emailVerified: true,
  name: "John Doe",
  username: "johndoe",
  // ... other fields
}
```

**Meteor Accounts (`users` collection):**
```javascript
{
  _id: "string-version-of-objectid",
  emails: [{ address: "user@example.com", verified: true }],
  profile: { name: "John Doe" },
  services: {},  // Empty - users must reset password
  username: "johndoe",
  // ... other fields preserved
}
```

## Safety Features

1. **Dry-run first** - Always previews changes before applying
2. **User confirmation** - Must explicitly approve migration
3. **Idempotent** - Safe to run multiple times (skips existing users)
4. **Error handling** - Exits immediately on any failure
5. **Verification** - Automatically verifies results after migration
6. **No data loss** - Source collection (`user`) is never modified
